# 스펙 기반 개발

한 줄짜리 수정에는 필요 없다. 한 세션을 넘기거나, 결정 사항이 유실될 수
있는 작업이라면 필요하다.

## specs/`<slug>`/ 폴더

비트리비얼한 기능마다 `specs/<kebab-case-slug>/` 디렉터리를 하나씩
만들고, 그 안에 파일 6개를 둔다.

| 파일 | 담는 내용 |
|---|---|
| `README.md` | 배경 — 이 작업이 왜 존재하는지, 어떤 문제를 푸는지 |
| `spec.md` | 요구사항 — "완료"가 무엇인지, 검증 가능한 형태로 |
| `plan.md` | 구현 계획 — 만들 순서대로의 단계 |
| `tasks.md` | 실행 가능한 체크리스트 — 진행하며 체크 |
| `findings.md` | 발견·결정·실수, 날짜와 함께 |
| `progress.md` | 재개 로그: 뭐가 끝났고, 다음은 뭐고, 뭘 다시 읽어야 하는지 |

## 스펙 시작하기

혼자 초안 쓰지 말고 먼저 인터뷰한다. 한 번에 한 질문씩. 사실은 직접
탐색하고(코드 읽기, 기존 컨벤션 확인), 결정만 사람에게 묻는다.
`spec.md`("무엇을")보다 `README.md`("왜")를 먼저 쓴다 — 배경이 명시되지
않은 스펙은 나중에 스코프가 슬금슬금 늘어나기 쉽다.

스펙에 합의되면, 코드를 건드리기 전에 `plan.md`(단계)와
`tasks.md`(계획에서 뽑은 구체적이고 체크 가능한 단계)를 쓴다.

## 구현하는 동안

결정, 기각한 대안, 실수가 생기는 그 순간 `findings.md`에 적는다 —
세션이 끝날 때 몰아서 적지 않는다. 각 항목: 날짜, 무슨 일이 있었는지,
왜 그랬는지, 뭘 기각했고 왜인지. 대안을 적어두지 않은 결정은 나중에
읽는 사람(다음 주의 나를 포함해서)에게 자의적으로 보인다.

세션을 끝내기 전, 오래 걸리는 작업 전, 또는 컨텍스트가 압축될 것 같다고
느껴질 때 `progress.md`를 갱신한다. 이 대화를 전혀 기억하지 못하는
사람이 읽는다고 가정하고 쓴다: 뭐가 끝났고, 다음은 뭐고, 어떤 파일이
중요하고, 뭘 다시 하면 안 되는지.

## 자기합리화 체크

| 생각 | 반박 |
|---|---|
| "이건 작아서 specs 폴더 없이 해도 됨" | 한 세션 안에 끝나고 기억해둘 결정도 없다면 정말 작은 게 맞을 수 있다 — 다만 구현 중간에 아니었다는 걸 알게 되기 전에, 미리 그 가정을 확인하라. |
| "findings는 끝에 몰아서 적자" | 끝에 가면 처음 두 접근을 왜 기각했는지 기억나지 않는다. 아직 명확할 때 적어라. |
| "progress.md는 진짜로 멈출 때 갱신하자" | 컨텍스트 압축이나 중단은 예고 없이 온다. 끝날 때만이 아니라 자연스러운 체크포인트마다 갱신하라. |
