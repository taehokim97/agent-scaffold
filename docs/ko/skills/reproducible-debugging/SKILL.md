# 재현 가능한 디버깅

재현할 수 없는 버그는 고칠 수 없다. 아래 4단계를 순서대로, 건너뛰지 않고
통과한다.

## Phase 1 — 재현

- 실패를 결정론적으로 재현하는 커맨드(테스트, 스크립트, curl 등) 하나를
  만든다.
- 재현이 안 되면 그게 지금 할 일이다 — "가끔 실패한다"는 상태로 다음
  단계로 넘어가지 않는다.
- 재현 커맨드와 실제 출력(에러 메시지, 스택트레이스, diff)을 기록한다.

## Phase 2 — 원인

- 재현된 실패를 이등분/격리해 원인을 좁힌다.
- 원인을 증거(로그 한 줄, 스택트레이스, 실패하는 diff)와 함께 한 문장으로
  말할 수 있어야 다음 단계로 간다. "아마 이것 때문일 것"은 원인이 아니다.

## Phase 3 — 수정

- Phase 2에서 말한 원인만 고친다. 발견한 김에 눈에 띈 다른 문제까지 같이
  고치지 않는다.

## Phase 4 — 검증

- Phase 1의 재현 커맨드가 이제 통과하는지 확인한다.
- 관련 테스트 스위트를 돌려 회귀가 없는지 확인한다.

## 자기합리화 체크

아래처럼 생각하고 있다면, 그건 단계를 건너뛰라는 신호가 아니라 단계를
지키라는 신호다.

| 생각 | 반박 |
|---|---|
| "이 버그는 뻔해서 재현 안 해도 바로 고칠 수 있다" | 뻔해 보이는 원인의 상당수가 실제로는 틀렸다. 재현 없는 수정은 검증되지 않은 추측이다. |
| "이미 원인을 알 것 같으니 증거 수집은 생략한다" | "알 것 같다"와 "증거로 확인했다"는 다르다. 증거 없이 넘어가면 Phase 3에서 엉뚱한 곳을 고친다. |
| "고치면서 발견한 다른 버그도 같이 고치자" | 스코프를 넘는 수정은 별도 이슈로 남긴다. |
